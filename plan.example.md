# [任务名] Plan

`plan.md` 是单文件规格驱动执行合同。把本模板复制到根目录 `plan.md` 后，替换成当前任务的真实内容。不要保留占位符。

## 1. 需求文档

写给用户看的非技术说明。

必须回答：

- 用户真正要解决什么实际问题。
- 谁会使用这个能力。
- 用户完成任务时应该看到什么体验。
- 当前范围包含什么。
- 当前范围不包含什么。
- 怎样算业务上完成。

要求：

- 写 WHAT 和 WHY。
- 不写框架、文件、函数、数据库、接口等实现细节。
- 不把用户建议的做法直接写成业务需求。

## 2. 当前事实

写已经确认的仓库事实。

必须覆盖：

- 当前代码事实。
- 当前测试事实。
- 当前文档事实。
- 当前配置事实。
- 当前命令输出事实。
- 当前已存在能力。
- 当前缺口。
- 当前未知点。

要求：

- 只写证据支持的事实。
- 用户偏好、外部项目和历史讨论只能作为证据来源。
- 未确认内容写成未知点。

## 3. 失败测试

先写什么会失败。

可以包含：

- 自动测试。
- 类型检查。
- 构建命令。
- CLI 或 UI 检查。
- 日志或状态检查。
- 真实用户路径演练。

要求：

- 每条失败测试必须对应真实产品行为或真实工程风险。
- 能自动测就自动测。
- 不能自动测就写清楚手动检查方式和期望结果。

## 4. 目标

写最终要交付的可验收结果。

必须回答：

- 用户路径完成到哪里。
- 代码主链路接到哪里。
- 状态和记录落在哪里。
- 输出呈现到哪里。
- 测试和文档同步到哪里。

要求：

- 可观察。
- 可验证。
- 可判定完成或失败。

## 5. 不做范围

写清楚当前任务不做什么。

要求：

- 不做范围必须服务边界清晰。
- 不写假兼容。
- 不写没有落点的未来设想。
- 不保留当前产品没有的能力入口。

## 6. 设计

写技术设计或文档设计。

必须覆盖：

- 主链路：输入 -> 判断 -> 状态 -> 执行 -> 输出 -> 记录。
- 模块边界。
- 文件职责。
- 状态归属。
- 数据或事件流。
- 错误、恢复、中断、重试边界。
- 与现有架构的关系。
- 影响的测试和文档。

不适用的章节要写明原因或删除。

## 7. 实施任务

写可执行 checklist。每项必须是动作，且能判断完成或未完成。

- [ ] T001 动作；文件或模块：；依赖：；验收：
- [ ] T002 动作；文件或模块：；依赖：；验收：
- [ ] T003 动作；文件或模块：；依赖：；验收：

## 8. 验证计划

写最后怎么证明完成。

必须包含：

- 相关局部测试。
- 完整验证命令。
- 构建或安装检查。
- CLI/UI/远程入口检查。
- 恢复或中断演练。
- 文档同步检查。
- 未验证内容。
- 剩余风险。

## 9. 收口

任务结束时更新。

必须回答：

- 目标是否完成。
- 失败测试是否变绿。
- 改了哪些文件。
- 跑了哪些验证。
- 哪些内容没有验证。
- 剩余风险是什么。
- 是否需要 commit 或 push。
